Doxygen Documentatie Guidelines

Dit document beschrijft hoe we Doxygen gebruiken voor API documentatie in SolarResult Dinnis Core.

ğŸ“‹ Filosofie

Onze documentatie is geschreven voor embedded developers met sterke technische achtergrond:

  • Begrijpen low-level concepten (memory layout, CPU cycles, state machines)
  • Willen design rationale, niet alleen "wat doet het"
  • Verwachten concrete cijfers (RAM/FLASH/timing)
  • Waarderen trade-off analyses

Kernprincipe: Module documentatie = <module>_doc.md

Voor de inhoud van module documentatie, zie Documentation Guidelines.

ğŸ“š Documentatie Types

We onderscheiden twee types documentatie met duidelijke naamgeving:

Type Doel Titel Formaat Voorbeeld
API Referentie Technische API details: functies, structs, parameters <Module> - API Referentie P1 - API Referentie
Design Documentatie Design rationale, specificaties, use cases, trade-offs <Module> - Documentatie P1 - Documentatie

Waarom deze scheiding?

Dit volgt het DiÃ¡taxis framework (industry best practice):

  • API Reference = "Wat doet deze functie?" (droge beschrijving)
  • Explanation = "Waarom werkt het zo? Wat was de gedachtegang?" (discussie)

    • Optioneel: Extra files zoals shell commands
