2. Doel en Verantwoordelijkheden

De App module fungeert als de systeemsupervisor. Het is een minimale module met twee cruciale verantwoordelijkheden die de algehele robuustheid van het systeem garanderen:

  1. Hardware Watchdog Management: De module initialiseert en beheert het task_wdt (Task Watchdog) systeem. Het voedt niet alleen zijn eigen watchdog-kanaal, maar fungeert ook als de centrale afhandelaar voor timeouts van alle andere modules. Als een module vastloopt, zorgt de App module voor een gecontroleerde herstart.
  2. System-wide ZBUS Monitoring: Voor debug-doeleinden bevat de module een ZBUS-monitor die luistert naar alle berichten op de ZBUS. Dit biedt een complete trace van de systeemactiviteit, wat essentieel is voor het opsporen van problemen in de communicatie tussen modules.

3. Functionaliteit

3.1 Hardware Watchdog

  • Initialisatie: Tijdens de SYS_INIT fase wordt de hardware watchdog geconfigureerd met een centrale callback-functie (task_wdt_callback).
  • Feeding Loop: De app_thread is een eenvoudige, laag-prioriteit thread die periodiek zijn eigen watchdog-kanaal voedt. Dit bevestigt dat de applicatie-loop nog draait.
  • Centrale Timeout Handler: De task_wdt_callback wordt aangeroepen als enige module in het systeem zijn watchdog niet op tijd voedt. Deze functie:
    1. Logt een foutmelding met de naam van de vastgelopen thread.
    2. Publiceert een bericht op de reboot_chan om een graceful shutdown aan te vragen.
    3. Als een tweede timeout optreedt voordat de herstart is voltooid, wordt een geforceerde, directe herstart van het systeem uitgevoerd.

3.2 ZBUS Monitor

  • De module definieert een ZBUS_LISTENER (zbus_monitor) zonder specifieke kanalen, waardoor deze naar alle ZBUS-kanalen luistert.
  • De zbus_monitor_callback functie logt de naam van elk kanaal waarop een bericht wordt gepubliceerd.
  • Dit is een krachtig diagnostisch hulpmiddel dat standaard is ingeschakeld in de debug-build.

4. ZBUS Interface

4.1 Gepubliceerde Kanalen

  • Geen. De App module publiceert zelf geen berichten.

4.2 Geobserveerde Kanalen

  • De zbus_monitor luistert naar alle kanalen voor logging.
  • De module heeft geen functionele afhankelijkheden van andere kanalen.

4.3 Legacy Kanalen

  • TRIGGER_CHAN: Dit kanaal wordt nog steeds gedefinieerd in app.c om compatibiliteit met andere modules te behouden die het declareren. Het is echter verouderd en wordt nergens meer gebruikt. De functionaliteit is vervangen door timing_chan.

5. Architectuur

5.1 Thread Model

  • EÃ©n thread: app_thread.
  • Prioriteit: Draait op de laagste applicatieprioriteit (CONFIG_APP_THREAD_PRIORITY_APP). Dit is een bewuste keuze omdat het voeden van de watchdog niet tijd-kritisch is en geen andere, belangrijkere taken mag verstoren.
  • Executie: Een simpele while(true) loop die elke CONFIG_APP_MODULE_WATCHDOG_TIMEOUT_SECONDS / 2 seconden de watchdog voedt.

5.2 Configuratie

De module wordt geconfigureerd via Kconfig.app:

  • CONFIG_APP_MODULE_WATCHDOG_TIMEOUT_SECONDS: De timeout-periode voor de watchdog van de App module zelf. De feed-interval is de helft van deze waarde.
  • CONFIG_APP_MODULE_LOG_LEVEL: Bepaalt het log-niveau voor de module.
  • CONFIG_APP_MODULE_THREAD_STACK_SIZE: Stack size voor app thread
    • Minimum: Klein (< 1KB), thread heeft minimale stack usage
  • CONFIG_APP_MODULE_LOG_LEVEL: Log level voor debug output

Watchdog Callback Behavior

Bij watchdog timeout (= app thread hangt):

  1. task_wdt_callback() wordt aangeroepen met:
    • channel_id: Watchdog channel ID
    • user_data: Pointer naar k_tid_t van app thread
  2. Callback loggeert thread naam en channel ID (voor postmortem analysis)
  3. TODO: Systeem reboot triggeren (momenteel alleen logging)

Rationale voor Delayed Reboot:

  • Current implementation loggeert alleen (voor development debugging)
  • Productie versie moet automatische reboot triggeren via Reboot Module
  • Zie Reboot Module documentatie voor graceful shutdown mechanisme

Native Sim Handling

Voor native_sim board (desktop testing):

  • HW_WDT_DEV wordt NULL (geen echte hardware watchdog)
  • Task Watchdog systeem werkt nog steeds (software-only timeout detection)
  • Geen hardware reset bij timeout (alleen callback logging)
