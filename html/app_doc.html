Waarom

Hardware Watchdog als Safety Mechanism

De Application Module fungeert als minimale coÃ¶rdinator met Ã©Ã©n primaire verantwoordelijkheid: het initialiseren en voeden van de hardware watchdog. Dit is een bewuste design keuze om system crashes te voorkomen zonder onnodige complexiteit in de main thread.

Rationale:

  • Scheiding van Concerns: Business logic zit in gespecialiseerde modules (p_one, energy_pulse, cloud), niet in main()
  • Fail-Safe Design: Als alle andere modules vastlopen, zorgt de watchdog voor automatische reboot
  • Minimale Overhead: De app thread doet niets anders dan watchdog feeding - geen message processing, geen state machines
  • Centraal Foutpunt: Watchdog timeout callback loggeert thread ID van vastgelopen module voor debugging

TRIGGER_CHAN als CoÃ¶rdinatie Mechanisme

De module definieert TRIGGER_CHAN als eigenaar van het trigger-moment, hoewel de trigger zelf door de Timing Module wordt gepubliceerd.

Rationale voor Channel Ownership:

  • Declaratie Locatie: Andere modules moeten TRIGGER_CHAN kunnen declaren zonder afhankelijkheid van timing module
  • Losere Koppeling: Modules subscriben op een "trigger event", niet specifiek op "timing module output"
  • Future-Proof: Als trigger logica verandert (bijv. externe interrupt), blijft de channel interface stabiel

Wat

Functionaliteit

De Application Module heeft twee taken:

  1. Hardware Watchdog Initialisatie (SYS_INIT fase)
    • Initialiseert Task Watchdog systeem met hardware watchdog device
    • Slechts Ã©Ã©n keer tijdens boot, POST_KERNEL prioriteit
    • Faalt met assertion als initialisatie mislukt (niet recoverable)
  2. Watchdog Feeding Loop (Runtime thread)
    • Voegt eigen channel toe aan task watchdog
    • Sleept CONFIG_APP_MODULE_WATCHDOG_TIMEOUT_SECONDS / 2 tussen feeds
    • Loggeert error en stopt thread bij feed failure (triggers watchdog reboot)

ZBUS Interface

Published Channels:

  • TRIGGER_CHAN: Bevat struct trigger_msg met timestamp_ms field
    • Ownership: App module definieert de channel (ZBUS_CHAN_DEFINE)
    • Publisher: Timing Module publiceert naar deze channel (niet de App module!)
    • Content: Timestamp van trigger moment (in milliseconds)
    • Subscribers: p_one, energy_pulse, message_queue (data acquisition modules)

Rationale voor Channel Ownership zonder Publishing: Dit lijkt contra-intuÃ¯tief, maar is een architecturale keuze:

  • App module is de eigenaar van de channel (definieert de interface)
  • Timing module is de producer van trigger events (publiceert data)
  • Hierdoor kunnen subscribers TRIGGER_CHAN declaren zonder timing_zbus.h include
  • Zie Timing Module documentatie voor trigger publicatie mechanisme

Subscribed Channels:

  • Geen: App module luistert niet naar andere modules

Hoe

Architectuur

Thread Model

EÃ©n thread: app_thread

  • Prioriteit: CONFIG_APP_THREAD_PRIORITY_APP (laagste prioriteit in systeem)
  • Stack Size: CONFIG_APP_MODULE_THREAD_STACK_SIZE bytes
  • Execution Model: Simpele while(true) loop, geen ZBUS listeners, geen state machine

Rationale voor Lage Prioriteit:

  • Watchdog feeding is niet time-critical (timeout is meerdere seconden)
  • Andere modules (p_one UART, cloud network) hebben hogere prioriteit nodig
  • Geen preemption risk voor critical operations

Initialisatie Volgorde

  1. SYS_INIT fase (POST_KERNEL): watchdog_init() configureert hardware watchdog
  2. Thread Start (K_THREAD_DEFINE): app_thread begint onmiddellijk na kernel start
  3. Runtime Loop: Thread voegt zich toe aan task watchdog en begint feeding cycle

Timing Afhankelijkheid: De app thread start altijd vÃ³Ã³r de Timing Module publiceert naar TRIGGER_CHAN. Dit is geen probleem omdat:

  • App module publiceert zelf niet naar TRIGGER_CHAN
  • Channel wordt alleen gedefinieerd (lege initialisatie: ZBUS_MSG_INIT(0))
  • Eerste publish gebeurt pas na Timing Module initialisatie (later in boot sequence)

Configuratie

Kconfig opties in Kconfig.app:

  • CONFIG_APP_MODULE_WATCHDOG_TIMEOUT_SECONDS: Watchdog timeout in seconden
    • Feed Interval: Helft van timeout waarde (safety margin)
    • Voorbeeld: 60s timeout â†’ feed elke 30s
  • CONFIG_APP_MODULE_THREAD_STACK_SIZE: Stack size voor app thread
    • Minimum: Klein (< 1KB), thread heeft minimale stack usage
  • CONFIG_APP_MODULE_LOG_LEVEL: Log level voor debug output

Watchdog Callback Behavior

Bij watchdog timeout (= app thread hangt):

  1. task_wdt_callback() wordt aangeroepen met:
    • channel_id: Watchdog channel ID
    • user_data: Pointer naar k_tid_t van app thread
  2. Callback loggeert thread naam en channel ID (voor postmortem analysis)
  3. TODO: Systeem reboot triggeren (momenteel alleen logging)

Rationale voor Delayed Reboot:

  • Current implementation loggeert alleen (voor development debugging)
  • Productie versie moet automatische reboot triggeren via Reboot Module
  • Zie Reboot Module documentatie voor graceful shutdown mechanisme

Native Sim Handling

Voor native_sim board (desktop testing):

  • HW_WDT_DEV wordt NULL (geen echte hardware watchdog)
  • Task Watchdog systeem werkt nog steeds (software-only timeout detection)
  • Geen hardware reset bij timeout (alleen callback logging)
