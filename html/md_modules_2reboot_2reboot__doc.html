Wat? (Functionaliteit & Data Flow)

Reboot Types

REBOOT_TYPE_FOTA (Firmware Over-The-Air update reboot):

  • Publish REBOOT_REQUEST naar alle modules
  • Wait voor SHUTDOWN_ACK van alle modules (max 10s timeout)
  • sys_reboot() call
  • Gebruikt na successful firmware download

REBOOT_TYPE_ERROR (Error recovery reboot):

  • Publish REBOOT_REQUEST naar alle modules
  • Wait voor SHUTDOWN_ACK van alle modules (max 10s timeout)
  • sys_reboot() call
  • Gebruikt bij recoverable errors die reboot vereisen

Reboot Redenen

Verschillende reboot redenen worden getrackt voor diagnostics:

  • UNKNOWN: Default na boot
  • USER: User requested (nRF Cloud command)
  • UPDATE: Firmware update
  • WATCHDOG: Task watchdog timeout
  • ERROR: Critical error
  • NETWORK: Network recovery

ZBUS Protocol

Published Channels:

  1. REBOOT_CHAN - Reboot request broadcast
    • Bevat: type (REBOOT_TYPE_FOTA/REBOOT_TYPE_ERROR), reason (waarom reboot), delay_ms (wachttijd)
  2. SHUTDOWN_REQUEST_CHAN - Request modules to shutdown
    • Empty message trigger

Subscribed Channels:

  1. SHUTDOWN_ACK_CHAN - Acknowledgement from modules
    • Bevat: module_name string identifier ("p_one", "cloud", etc.)

Acknowledgement Tracking

Completion Check: Module compares aantal ontvangen ACKs met aantal observers van SHUTDOWN_REQUEST_CHAN.

Logic:

  • Query ZBUS voor enabled observer count van SHUTDOWN_REQUEST_CHAN
  • Compare met aantal ontvangen SHUTDOWN_ACK messages
  • Als ack_count >= observer_count â†’ alle modules hebben gerespondeerd â†’ safe to reboot

Waarom observer count?:

  • Telt hoeveel modules subscribed zijn op SHUTDOWN_REQUEST
  • Als module disabled is (Kconfig), wordt deze niet meegeteld
  • Dynamic: Werkt met elke module configuratie

Timeout Mechanisme

Probleem: Module kan hangen tijdens shutdown (bijv. UART read blocked)

Oplossing: Watchdog timer van 10 seconden start bij shutdown request. Als niet alle ACKs ontvangen binnen timeout, force reboot anyway.

Timeout handler gedrag:

  • Log WARNING met aantal missing ACKs
  • Submit reboot work (force shutdown)

Trade-off:

  • âœ… Voorkomt indefinite hang bij module failure
  • âš ï¸ Risk: Module data loss als shutdown niet af is binnen 10s
  • Balance: 10s is ruim (normale shutdown < 2s)

Work Queue Pattern

Probleem: sys_reboot() is blocking call, mag niet in ZBUS callback uitvoeren (deadlock risk).

Oplossing: Schedule reboot via dedicated work queue item.

Rationale:

  • ZBUS callbacks moeten snel return (non-blocking requirement)
  • sys_reboot() blokkeert indefinite (geen return)
  • Work queue draait in eigen thread context (geen ZBUS blocking)
  • Delay parameter (delay_ms) kan via k_work_schedule() timing

Module Shutdown Implementatie Vereisten

Wat modules moeten doen bij shutdown:

  1. Stop actieve operaties
    • Disable UART/GPIO interrupts
    • Cancel pending timers
    • Stop background threads (waar mogelijk)
  2. Persist kritieke state
    • Save rolling statistics naar flash (Settings API)
    • Commit database transactions
    • Flush pending writes
  3. Cleanup resources
    • Free allocated memory (waar applicable)
    • Clear sensitive buffers
    • Close open connections (network sockets)
  4. Send ACK
    • Publish SHUTDOWN_ACK met module naam
    • Timing: binnen 10 seconden (shutdown timeout)

Voorbeeld flow (P1 module):

  • Stop UART reception
  • Save rolling stats naar Settings + commit
  • Clear RX buffers
  • Publish ACK("p1")
  • Total time: ~500ms

Checklist voor nieuwe modules:

  • Subscribe to REBOOT_CHAN
  • Implement cleanup handler
  • Flush buffers
  • Save persistent state
  • Close connections
  • Send SHUTDOWN_ACK
  • Test: shutdown completes < 10s
